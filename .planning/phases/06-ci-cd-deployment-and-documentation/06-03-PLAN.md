    1. **Folder layout** — Show the actual `src/app/core/store/` tree with one-line descriptions per file. Include the file names as they actually exist.

    2. **Design rationale for each layer:**
       - `actions/` — 15 action creators following command/event naming: 5 commands (`loadTasks`, `addTask`, `moveTask`, `updateTask`, `removeTask`) + 10 event counterparts. Naming signals intent: commands are imperative, events are past-tense outcomes.
       - `reducers/` — `createFeature` auto-generates the feature selector, removing boilerplate. `EntityAdapter` provides normalised O(1) CRUD — no hand-written array mutations. Optimistic `moveTask` applies state immediately; `moveTaskFailure` reverts via `previousColumnId` in the action payload (not a second selector call).
       - `selectors/` — Factory selector pattern (`(columnId) => createSelector(...)`) over the deprecated NGRX `props` pattern. `selectCompletionRate` guards against division-by-zero at the selector level (returns 0 for empty task list). `selectCountByPriority` pre-initialises all 4 Priority keys to 0 so consumers never need key-existence guards.
       - `effects/` — `concatMap` for all mutation effects (prevents request cancellation and reordering). `catchError` inside the inner `concatMap` pipe (not the outer) so the effect stream survives individual errors. `inject()` fields declared before `createEffect()` fields — required for Vitest DI safety (NGRX issue #4708).
       - `index.ts` — Named barrel export `{ TaskEffects }` prevents `@ngrx/effects` internals from bleeding through to consumers.

    3. **Key constraints** — `TaskState.error` is `string | null` (non-optional) — `createFeature` auto-selector requirement. `taskAdapter` exported separately from `tasksFeature` so Plan 04 selectors can call `adapter.getSelectors()`.

    **docs/interview-talking-points.md** (DOC-07):

    Structure by the exercise evaluation criteria. Write as actual talking points the candidate would deliver verbally — specific, reference-based, not generic.

    Sections:

    **1. State Management**
    - EntityAdapter: normalised store, O(1) CRUD, avoids array mutation boilerplate
    - Optimistic `moveTask`: the `previousColumnId` is in the action payload — rollback does not require a selector call in effects (contrast: if it queried the store in the effect, there's a race condition window)
    - `concatMap` vs `switchMap`: mutations must not be cancellable. A second move request cannot silently cancel an in-flight first move. `concatMap` serialises them.
    - `catchError` placement: inside the inner pipe, not the outer — the effect stream must survive individual errors without terminating

    **2. Signals Usage**
    - `input()` signals on `TaskCardComponent` — the task arrives as a signal, not a plain `@Input()`. Downstream `computed()` chains react automatically.
    - Computed signal chain: `task()` → `priorityClass()` (CSS class string), `formattedDate()` (display string), `isOverdue()` (boolean from `createdAt` + `OVERDUE_THRESHOLD_DAYS`)
    - `store.selectSignal()` bridge: at the `BoardPageComponent` boundary, NGRX observable selectors are converted to Angular signals. No async pipes, no manual subscriptions in smart components.
    - `signal()` not `model()` for `isExpanded`/`isEditMode`: `model()` is for two-way parent binding. Local UI-only state uses `signal()`. Using `model()` here would be semantically wrong.

    **3. Dynamic Component Rendering**
    - `ViewContainerRef.createComponent()` creates the `ComponentRef`; `ComponentRef.setInput()` passes typed inputs
    - Observable inputs: subscribed internally, value pushed via `setInput()` on each emission, subscription tracked for cleanup
    - Signal inputs: wrapped in `effect()` that calls `setInput()` on each signal change. The `effect()` is wrapped in `untracked()` to avoid Angular 21 NG0602 — nested `effect()` in a reactive context is prohibited
    - `DestroyRef` + `takeUntilDestroyed()` cleans all subscriptions on directive destroy; `ComponentRef.destroy()` tears down each rendered component

    **4. Architecture**
    - Smart/dumb split: `BoardPageComponent` holds all store wiring. `ColumnComponent` and `TaskCardComponent` receive plain signal inputs, emit output events. Completely decoupled from the store — testable in isolation.
    - `core/` / `features/` / `shared/` organisation: `core/` is singleton services and state; `features/board/` is the single feature; `shared/` is the dynamic widget system (reusable, store-independent)
    - Factory selectors over props pattern: `selectTasksByColumn` returns `(columnId: string) => createSelector(...)`. Called once at component field initialisation — selector recreation defeating memoization is prevented.

    **5. Code Quality**
    - Discriminated union `_exhaustiveCheck` in `TaskAction`: compile-time proof that every variant is handled. TypeScript's `never` type makes missing cases a compile error, not a runtime bug.
    - `export type` for interfaces: required for `isolatedModules` (TS1205). All model interfaces use `export type`.
    - `inject()` before `createEffect()`: Vitest DI safety — NGRX issue #4708. Field initialisation order matters for TestBed DI.
    - `provideStoreDevtools({ maxAge: 25, autoPause: true })`: performance-safe DevTools config. `autoPause` prevents DevTools from slowing the app when the panel is not open.

    **docs/gsd-framework-notes.md** (DOC-08):

    Explain GSD clearly for two audiences: (1) the interviewer who wants to understand how AI was used, (2) someone curious about structured AI-assisted development.

    Content:

    **What is GSD?**
    GSD (Get-Shit-Done) is a structured AI planning framework that externalises intent before writing code. It runs as a series of slash commands in Claude: `/gsd:discuss-phase` (capture intent), `/gsd:research-phase` (domain research), `/gsd:plan-phase` (task decomposition), `/gsd:execute-phase` (implementation), `/gsd:verify-work` (acceptance testing).

    **Why it was a good choice for this exercise:**
    - Prevents scope creep: each phase produces a CONTEXT.md with locked decisions. The executor cannot invent features that were deferred.
    - Creates a verifiable audit trail: every architectural decision in `STATE.md` was explicitly approved before implementation. The interviewer can trace any decision back to a recorded rationale.
    - Matches the exercise's AI transparency requirement: GSD makes AI assistance structured and auditable, not opaque. The planning files are committed alongside the code.
    - Natural dependency chain: GSD's phase decomposition (foundation → store → components → widgets → testing → docs) matched the Angular application's natural dependency chain. This wasn't coincidence — GSD forces you to think about what must exist before what.
    - Quality gate: each phase has a `/gsd:verify-work` step that checks against acceptance criteria. This caught integration issues (e.g., the Storybook 10 Angular builder requirement) before they became bugs.

    **What GSD produced for this exercise:**
    - 6 phases, 17 execution plans
    - Research documents for high-risk phases (tooling compatibility, ViewContainerRef patterns)
    - Task-level plans with explicit `<verify>` criteria — the executor cannot complete a task without a passing automated check
    - `STATE.md` accumulating every decision across all phases — the single source of truth for "why"

    **How it compares to unstructured AI usage:**
    Without GSD, AI assistance tends toward "vibe coding" — accepting generated code without understanding the constraints. GSD forces the developer to specify intent first, review research, and approve every task before execution. The output is code the developer can defend line by line.

    **docs/data-flow-diagram.md** (DOC-09):

    Write a Mermaid `sequenceDiagram` showing the `moveTask` optimistic update flow. This is the most interview-worthy flow in the codebase — it shows optimistic state update, effect coordination, and rollback in one diagram.

    Use the exact diagram from the research file (verified against actual implementation):

    ```mermaid
    sequenceDiagram
        participant U as User (select box)
        participant C as BoardPageComponent
        participant S as NgRx Store
        participant R as task.reducer
        participant E as TaskEffects
        participant M as TaskMockService

        U->>C: column select change
        C->>S: dispatch moveTask({ taskId, columnId, previousColumnId })
        S->>R: moveTask action
        R-->>S: state update (task.columnId = columnId) IMMEDIATELY
        S-->>C: selectSignal() emits — task moves visually
        S->>E: moveTask$ effect picks up action
        E->>M: moveTask(taskId, columnId)
        alt success
            M-->>E: Observable completes
            E->>S: dispatch moveTaskSuccess()
            Note over R: no-op (already applied)
        else failure
            M-->>E: throwError()
            E->>S: dispatch moveTaskFailure({ taskId, previousColumnId })
            S->>R: moveTaskFailure action
            R-->>S: state rollback (task.columnId = previousColumnId)
            S-->>C: selectSignal() emits — task moves back
        end
    ```

    Add explanatory prose before the diagram covering:
    - Why this flow is interesting: the reducer handles `moveTask` synchronously (zero latency to the user) while the effect runs asynchronously. The effect is the eventual-consistency check, not the primary state writer.
    - The `previousColumnId` design: carried in the action payload so the reducer can roll back without querying the store in the effect (which would introduce a read-after-write race condition window).
    - The `moveTaskSuccess` no-op: the reducer ignores it because the optimistic state is already correct. The event exists to signal "the server confirmed" for potential audit logging or analytics.
    - Test entry point: `?failNextMove=1` in the URL sets `TaskMockService.shouldFail = true` directly — no DI override needed because the service is `providedIn: 'root'`. The Playwright E2E test uses this seam.

    Add a second, simpler diagram for the addTask flow (no optimistic update — straightforward effect with success/failure paths) to show the contrast.

    Mermaid syntax rules for GitHub compatibility:
    - Use `sequenceDiagram` type (most stable)
    - Keep participant labels short
    - No special characters in labels
    - Use `alt`/`else`/`end` for branching (not `opt` for optional)
    - Wrap the diagram in a fenced code block with `mermaid` language tag
