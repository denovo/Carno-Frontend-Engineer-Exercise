    ## Architecture Decisions  [DOC-02]

    Write substantive rationale for each decision (not just "what" but "why"). Source content from `STATE.md` accumulated context. Cover all of these:

    1. **Angular 21 (not 17 per spec)** — New file naming (`app.ts` not `app.component.ts`), signals are first-class, `input()` / `model()` APIs. Still meets spec intent.
    2. **NGRX with EntityAdapter** — Normalised O(1) CRUD, devtools support, `createFeature` auto-generates feature selector. Avoids hand-rolled immutable updates.
    3. **Optimistic `moveTask` with `previousColumnId` in payload** — State updates immediately (zero latency to user). Rollback without a selector call in effects — `previousColumnId` travels in the action, not retrieved after the fact. (`Phase 02-ngrx-store` decision)
    4. **`concatMap` for mutation effects** — Prevents request cancellation (vs `switchMap`) and reordering. A second move cannot cancel an in-flight first move. (`Phase 02-ngrx-store` decision)
    5. **`signal()` over `model()` for local UI state** — `isExpanded` and `isEditMode` have no parent two-way binding. Using `model()` would be wrong semantics. (`Phase 03-01` decision)
    6. **`store.selectSignal()` bridge** — Smart components bridge NGRX to Angular signals at the boundary. Presentational components receive plain `input()` signals — zero store coupling in leaf components. (`Phase 03` decision)
    7. **Smart/dumb component split** — `BoardPageComponent` (smart) holds all store wiring and passes signal values down via `input()`. `ColumnComponent`, `TaskCardComponent` are fully presentational. (`APP-08`)
    8. **`DynamicWidgetOutletDirective` with `untracked()`** — Angular 21 NG0602 prohibits nested `effect()` in a reactive context. The config effect wraps `renderOne()` in `untracked()`. (`Phase 04` decision)
    9. **Vitest via `@analogjs/vitest-angular`** — Jest-compatible API, faster than Karma, native ESM. Must run via `ng test --watch=false` (not bare `vitest run`) to get the Angular compiler plugin and `@app/` alias. (`Phase 03-01` decision)
    10. **Storybook 10 (not 8)** — `@storybook/angular@8` requires Angular <20. v10 supports Angular <22. Requires `@angular-devkit/build-angular` as peer dep and runs via `ng run petello:build-storybook`. (`Phase 05-02` decision)
    11. **`npm ci --legacy-peer-deps` everywhere** — Storybook 10 peer deps conflict with Angular 21. Required in CI and Vercel install command. (`Phase 05-testing-and-storybook` decision)

    ## Scalability Considerations  [DOC-03]

    Address the four scenarios the exercise spec explicitly calls out:

    1. **Multiple boards** — The `selectTasksByColumn` factory selector is already parameterised by `columnId`. Extending to multi-board requires a `boardId` key on the store slice and a `loadTasks({ boardId })` dispatch from a board-selection route. The mock service's interface already models this boundary.
    2. **Real-time collaboration** — A WebSocket effect alongside the HTTP effect could push `moveTaskSuccess` / `addTaskSuccess` directly when another user mutates data. The reducer handles these events already — no reducer changes needed. Conflict resolution (last-write-wins vs OT) is the only new concern.
    3. **Undo/redo** — `moveTask` already carries `previousColumnId` (designed with undo in mind). An undo stack reducer wrapping the existing store state transitions would complete the pattern. NGRX's `@ngrx/store` supports meta-reducers for exactly this.
    4. **Offline support** — Optimistic updates are already in place — the UI never waits for the server. Adding a service worker + an action queue (retry on reconnect) extends the existing `shouldFail` / rollback seam. The rollback path is already tested via the `?failNextMove=1` Playwright seam.
